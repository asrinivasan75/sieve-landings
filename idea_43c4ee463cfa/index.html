THE PM ARCHETYPES WE'VE ALL BEEN
ARCHETYPE_01 / THE OPTIMIST
Every launch has a "we'll monitor it closely" phase. Monitoring doesn't prevent the incident — it just tells you how bad it got.

You've run the sprint. The QA is green. The demo killed in the all-hands. There is no risk doc because everyone agreed it was low risk. Two weeks post-launch, you're in a bridge call explaining why checkout conversion dropped 11 points on iOS 17.

ARCHETYPE_02 / THE SPREADSHEET HERO
A 5x5 risk matrix scored by people who are emotionally invested in shipping is not a risk assessment. It's a consensus document in disguise.

Column A: risk. Column B: likelihood (1-5). Column C: impact (1-5). Column D: mitigation. All cells filled. All cells wrong. The failure modes that actually fire are the ones nobody thought to add as rows.

ARCHETYPE_03 / THE POST-MORTEM REGRET
Every post-mortem has a section called "what we'd do differently." That section is just a pre-mortem written six weeks too late.

The document is well-written. The contributing factors are clear. The action items have owners. And somewhere in the timeline you see a comment from three months ago that read "should we test this under load?" Nobody answered it.

FAQ
Reasonable concerns
01 If this is just pattern-matching against public post-mortems, why wouldn't I get the same output from ChatGPT? +
That's the honest version of the most important question to ask us. The short answer is: with a thin corpus, you'd be right. Public GitHub post-mortems and engineering blogs skew heavily toward infrastructure failures — they almost never capture product UX failures, support flood patterns, or compliance gaps like the GDPR consent issue above.
The differentiation we're building is a corpus of product-layer and UX failure modes that are almost never published — sourced through direct partnerships, structured beta user contributions, and a concierge-first data collection phase. The beta is specifically designed to test whether the pattern quality beats a senior PM's intuition. If it doesn't, we'll tell you before you pay anything.
We've set an explicit kill criterion on this: if fewer than 3 of 10 beta users report forecasted failure modes they hadn't already considered, we stop. We're not interested in building a prettier wrapper around something free.
02 I'm already under deadline pressure. Why would I run a tool that might surface more blockers right before launch? +
This is a real behavioral tension we haven't fully resolved yet, and we're not going to pretend otherwise. Optimism bias is structural — it doesn't go away because a tool exists.

What we've seen in early testing: the forecast's value shifts depending on timing. Run it during sprint planning, it becomes a scoping tool. Run it a week before launch, it becomes a "here's what to monitor closely" brief. Run it the day of launch, it becomes the post-mortem you write in advance.

The artifact is shareable. That matters. When the PM can hand their VP a ranked failure forecast instead of a blank risk doc, the political dynamic around surfacing risks changes. We're not naive about this — we're specifically interviewing PMs who've shipped features that caused support floods to understand exactly when and how they'd want to use this.
03 What happens if every feature gets a 20-item failure list? Won't people start ignoring it the same way they ignore linter warnings? +
Linter warning fatigue is exactly the failure mode we're trying to avoid, and it's an explicit design constraint on the output format.

The forecast is ranked, not exhaustive. We surface the top failure modes by severity × likelihood × historical match count, not every theoretical edge case. The goal is 5–8 high-signal items that feel specific to your feature — not a comprehensive taxonomy of everything that's ever gone wrong in software.

If users rate the output quality below 3.5/5 on average, that's a kill criterion we've committed to. The signal-to-noise ratio is the product. We'd rather ship fewer, sharper failure modes than pad a list to look thorough.
